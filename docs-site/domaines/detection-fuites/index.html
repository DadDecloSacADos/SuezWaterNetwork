Détection & Corrélation de fuites¶

Objectif de cette page

Comprendre comment AQWN détecte, localise et gère les fuites d'eau dans le réseau.

Prérequis : Hydraulique | Glossaire

Contexte métier¶

La détection de fuites est le cas d'usage principal d'AQWN. Un réseau d'eau perd typiquement 15-30% de son volume. Les fuites :

  • Coûtent cher : Eau perdue = perte de revenu + surcoûts énergie
  • Sont difficiles à détecter : Fuites invisibles sous terre jusqu'à rupture catastrophique
  • Doivent être localisées rapidement : Pour réduire le délai intervention

1. Base de données fuites¶

Entités & Attributs¶

Chaque fuite enregistre :

Attribut Type Obligatoire Notes
ID Texte ✅ Identifiant unique
Type Select ✅ Main / Branchement client
Localisation X/Y ⚠️ Soit X/Y soit adresse
Adresse Texte ⚠️ Géocodée → X/Y si absent
DMA/Zone Référence ✅ Zone hydraulique concernée
Date apparition Date ✅ Quand suspect commence
Commentaire apparition Texte ✅ Comment détecté ?
Date détection Date ✅ Quand confirmation
Commentaire détection Texte ✅ Preuve détection
Volume estimé Nombre ⚠️ Débit ou débit journalier
Date réparation Date ⚠️ Quand réparé
Date fermeture Date ⚠️ Quand dossier clôt
Label Select ✅ Fuite confirmée / Faux positif
Caractérisation Texte ⚠️ Détails type (rupture ? joint ?)
Intervention liée Référence ⚠️ Lien vers CIS/tâche d'intervention

Gestion des fuites¶

Création : - Automatique via détection (algorithme) - Manuelle via interface (opérateur signale) - Via événement SEF (Sector Event Flow)

Modification : - Changement statut (apparition → détection → réparation) - Affinement localisation - Mise à jour volume estimé

Fermeture : - Après intervention - Validation opérateur

2. Détection via corrélation de données (Data Correlator)¶

Qu'est-ce que le Data Correlator ?¶

Le Data Correlator est un module d'analyse statistique qui :

  • Détecte les changements abrupts (spikes) dans les données
  • Corrèle les changements entre plusieurs capteurs
  • Identifie la cause probable d'un événement

Exemple :

Pression passe de 2.5 bar à 1.8 bar à 14:30
  + Débit à la sortie zone passe de 50 L/s à 65 L/s
  + Logger acoustique détecte bruit à 14:31
  → Corrélation = Fuite probable dans la zone

Configuration¶

Paramètre Description Défaut
Delay (fenêtre) Durée recherche correlation 15 min
ThresholdFactor Amplitude min de variation Dataset-dépendant
Type corrélation Libre ou conditionnelle Libre
Allocation area Zone/DMA à analyser Global

Input du Correlator¶

Données passives : - Événement (ID, dates début/fin, zone) - Données capteurs (CSV : temps, valeur, validité) - Paramètres de recherche

Données actives (API) : - Contrat + plage temporelle - Récupération auto des séries disponibles

Output¶

  • Données corrélées : Liste des capteurs affectés + timing
  • Scores corrélation : Force du lien (0-100%)
  • Diagnostic : Type probable de fuite
  • Visualisation : Graphiques des courbes superposées

3. Localisation de fuite LLI¶

LLI = Localisation de Fuite par Indice¶

La LLI utilise des loggers acoustiques (capteurs de bruit) pour triangulariser la position d'une fuite.

Principe¶

Logger 1 (Acoustic)        Logger 2
    │ détecte bruit           │ détecte bruit
    │ t=0ms                   │ t=+3ms
    │                         │
    └────────────────────────┘
         Δt = 3ms

    Δ distance = Δt × c_sound ≈ 1 mètre
    → Fuite à ~1m de Logger 1

Configuration¶

  • Placementer des loggers sur le réseau (jonctions stratégiques)
  • Calibration fréquence + seuils
  • Synchronisation temps entre loggers
  • Validation géolocalisation

4. Fuites après compteurs¶

Cas particulier¶

Les fuites en aval du compteur client sont du ressort du client, mais AQWN peut :

  • Détecter anomalies (pression anormalement basse chez client)
  • Alerter le service client
  • Tracer historique des événements

Domaine dédié¶

Voir Fuites après compteurs pour détails.
